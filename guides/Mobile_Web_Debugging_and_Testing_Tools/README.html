Mobile Web Debugging and Testing Tools

Objective

In this section, you will learn the basics of using various web debugging tools for troubleshooting Titanium projects on Mobile Web.

Contents

The purpose of this document is to provide tips and insight on how to use the development tools provided by browsers to debug Titanium Mobile Web apps. Learning how to use these tools is not covered by this article, as the major browser manufacturers already have detailed instructions on their use. This document assumes that the reader is at least somewhat familiar with web development.

Important Concepts

Mobile Web UI

The implementation of UI elements in Mobile Web is different from the implementation of elements in the native platforms because a) controls in Mobile Web are almost always composites of other controls and b) layouts are always asynchronous.

Controls are almost always a collection of of other Titanium controls. This is important to remember when inspecting elements on a webpage because you will see more controls on the page than you created. As an example, a button contains a view with its layout set to horizontal. This view contains a Ti.UI.ImageView and a Ti.UI.Label, which are themselves composed of other elements. All together, a button contains 5 DIVs at minimum, and can contain even more. Understanding how these controls work under the hood is not necessary for debugging Titanium apps, but it is important to simply know that these "hidden controls" exist. To help with debugging, every Ti.UI.View has a property called widgetId that contains a unique identifier. For example, the first window that is opened has a widget ID of "Ti.UI.Window:0".

Mobile Web doesn't have the concept of heavy-weight/light-weight windows like the native platforms do. All windows are just DIVs that are attached to the page. Each top-level window is attached to a hidden root Ti.UI.View element that is persistent. This view is assigned a widgetID of "Ti.UI.View:0".

Another aspect of the Mobile Web UI mechanism that differs from the native platforms is that layouts are always asynchronous. To illustrate this point, consider the following code:

var win = Ti.UI.createWindow({
backgroundColor: "red"
});
win.open();
var button1 = Ti.UI.createButton({
title: "I'm a button"
});
win.add(button1);
button1.left = 10;
var button2 = Ti.UI.createButton({
title: "I'm also a button"
});
win.add(button2);
button2.left = 120;
console.log("done");

The left values of button1 and button2 are set, which will trigger a layout. The actual layout doesn't occur until after the current thread of execution has ended, which is after the console.log() call. If you were to set a breakpoint at the console.log("done") call, the value read back from button1.left will be 10, but the value in button1.rect.x will not be 10. This behavior must be kept in mind both when writing code and when setting breakpoints.

Layouts are performed asynchronously so that multiple layout updates can be batched together. A side effect is that calling view.updateLayout() doesn't provide any performance benefit on Mobile Web, unlike iOS and Android.

Networking

Networking in Mobile Web is limited by the Same Origin Policy. It is critical that you understand what the Same Origin Policy is and how it affects networking requests. To quote Mozilla:

The same origin policy prevents a document or script loaded from one origin from getting or setting properties of a document from another origin.

In short, the Same Origin Policy prevents you from doing any request from a domain other than the domain hosting the Mobile Web app. If you are developing using Studio's built in web server, this means that requests to anything other than http://localhost:8020 will fail! This includes requests to other ports on localhost. It's important to note that this behavior is an intentional security mechanism intended to prevent Cross Site Scripting Attacks and Cross Site Request Forgeries.

Browsers and servers implement a mechanism called Cross-Origin Resource Sharing (CORS) to enable legitimate cross-site requests to be made. A server hosting content meant to be accessed from a different domain adds the Access-Control-Allow-Origin header to responses that gives clients a white-list of domains that are allowed to access this resource. If the server sets the headers properly, then nothing needs to be done on the client. If the server does not support CORS, then a proxy will need to be used in conjunction with a URL formatter, as described in the API Documentation.

It's important to note that CORS violations do not throw an exception! Things will simply appear to "not work." If your networking requests doesn't appear to be working, the first thing you should do is check to see if a CORS violation is occuring.

Facebook

Facebook deserves a special mention here because there is a gotcha on Mobile Web that doesn't apply to the other platforms. When developing on Mobile Web, you must set the "Site URL" to point to the domain hosting your app. When developing, this means you must set it to localhost:8020, or whatever your local dev server is configured as.

IMPORTANT: If you do not set the Site URL correctly, then Facebook will not work correctly. No errors are thrown, but no results are returned either. It is a very silent failure.

WebViews

Mobile Web implements Ti.UI.WebView by using an iFrame. The Same Origin Policy affects web views as well as HTTP requests. When a source is set on the web view that is different than domain hosting the app, the app cannot set any properties inside of the web view.

It is also possible for servers to configure the X-Frame-Options header to prevent the site from being displayed in the web view (this problem is unique to Mobile Web). If this header is set, the web view will not load by design and will throw an error. Notably, Google sets this option for all of their pages (including their home page). This means that you cannot set the source of a web view to www.google.com!

WebKit Inspector

The WebKit Inspector is a development tool included with desktop WebKit-based browsers such as Chrome and Safari. If you are using Firefox or Internet Explorer to debug your app, then the information presented here will need to be mapped to the appropriate sections of Firebug and the F12 tools, respectively. This section will discuss how to use the inspector to debug Titanium apps in specific. For general information about using WebKit Inspector, refer to the Chrome Developer Tools documentation and the Safari Developer Tools documentation (they are mostly the same).

Elements Tab

The Elements tab shows the element hierarchy on the web page. Each element has a property called data-widget-id that shows the assigned widget ID. This can be used to correlate DIVs and other elements in the hierarchy with Titanium controls in JavaScript. You can right click on any element on a page and select "Inspect Element." There are plans to make this process easier in the future (TIMOB-9728).

Console Tab

This tab should be your first stop when you encounter a problem. All of the logging that the native platforms send to the console window in Studio is sent here in Mobile Web. All calls to Ti.API.* and console.* are directed here, and any exceptions encountered are logged here. Networking issues, such as CORS violations, are also logged here.

Network Tab

The Network tab shows all network connections the browser has made, when they were made, and how long they took. All requests, including images, script files, and the index.html page are shown.

Applications that take a long time to load in Mobile Web are almost always hung up waiting for files to download. The network tab can help you pinpoint scripts/images that are slowing down the app. There are lots of tricks to reducing transfer times, such as compressing images, performing production builds of your app which minifies code, deferring downloading scripts/images until they are needed (or pre-loading them, as the case may be), etc.

Instrumentation

Mobile Web has built-in instrumentation that provides developers with information on the state of an app. It can be enabled by adding the following code to the <mobileweb> section of your tiapp.xml file (this section may need to be created manually).

<instrumentation>true</instrumetation>

Instrumentation lists the following pieces of information:

  • System load time

  • Layout time

  • Network access time

System load time is the amount of time that elapsed from when the page first starts to load until the time that app.js is invoked. This time represents part of the time that users must wait until they see their first page (the rest of the time is spent in app.js and doing subsequent layouts). System load time is almost purely dependent on the size of the app (app,js, precached images, and precached scripts) and the user's network speed.

Bringing it all together

Using these tools together to solve problems is key to successfully debugging Mobile Web applications. Consider the following source code:

var win1 = Titanium.UI.createWindow({
title:'Tab 1',
backgroundColor:'#fff'
});
win1.open();
var i, j, view;
for (i = 0; i < 6; i++) {
for (j = 0; j < 6; j++) {
view = Ti.UI.createView({
width: 30,
height: 30,
left: i * 50,
top: j * 50,
borderWidth: 1,
borderColor: "#000"
});
view.backgroundColor = "#00" + (i + j);
win1.add(view);
}
}

This code will create a 6 x 6 grid of views and attach them to the page. The background views create a linear gradient from black to blue, starting at the upper left view and ending at the bottom right view. There is a bug in the code though: the lower right square is completely white! Why is that?

We start by finding the widget ID of the offending view by right clicking on it and inspecting the element.

We find that the widget ID is "Ti.UI.View:36". We then use this value to set a conditional breakpoint (right click on the breakpoint and select "Edit breakpoint") in the for loop so that it will break on this specific view (stepping through 36 views is tedious!).

After reloading the page (the breakpoint is persistent), we inspect the view's background color by hovering our cursor over the view.backgroundColor text.

Aha! The last element sets its background color to "#0010", which isn't a valid CSS color. Since i + j equals 10, we need to string cast the value as a hex number, not a decimal number:

var win1 = Titanium.UI.createWindow({
title:'Tab 1',
backgroundColor:'#fff'
});
win1.open();
var i, j, view;
for (i = 0; i < 6; i++) {
for (j = 0; j < 6; j++) {
view = Ti.UI.createView({
width: 30,
height: 30,
left: i * 50,
top: j * 50,
borderWidth: 1,
borderColor: "#000"
});
view.backgroundColor = "#00" + (i + j).toString(16);
win1.add(view);
}
}

With this change in place, everything works perfectly!

Debugging On Device

Once the application is working well, it's time to take the application to devices. For the most part, everything should work the same in mobile browsers as they do in desktop browsers, just much slower. Desktop browsers tend to run around 10 times faster than their mobile counterparts. Debugging is difficult though because most devices only provide a JavaScript console. Below are three ways to debug applications on Android, iOS, and Blackberry. Unfortunately there is no way to debug Windows Phone 7.5 web apps other than using alert()'s.

Enabling the JavaScript Console in Android:

Most Android devices can only be debugged using ADB logcat, which requires the device to be connected to a PC. You can filter the results of adb logcat to only show browser logging by piping the command to grep (OS X and Linux only) like so:

adb logcat | grep "^..browser"

Some Android 3 and newer browsers also support enabling a console on device by typing "about:debug" into the address bar after the page has loaded.

Enabling the JavaScript Console in iOS:

iOS devices can be debugged by enabling the Debug Console in Mobile Safari's Advanced settings.

Using the BlackBerry Remote Web Inspector:

BlackBerry 7 and Playbook devices can be debugged remotely using a desktop browser via WebKit Inspector. The full details on using it are described Debugging BlackBerry web applications using Web Inspector on the BlackBerry Developer Blog.

WEINRE

Another option for remotely debugging apps is to use WEb INspector REmote. WEINRE provides middleware that enables WebKit inspector to remotely debug apps, similar to how BlackBerry does it. There are a few limitations in WEINRE, such as the inability to set breakpoints, but it is still more powerful than just using a JavaScript console.
