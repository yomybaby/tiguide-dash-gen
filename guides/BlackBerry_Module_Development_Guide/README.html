Introduction

Titanium provides the ability to extend the built-in functionality of the Titanium SDK through a series of optional extensions called modules. Modules let you write code in a platform's native language (C++ for BlackBerry). Through a series of classes and attributes, we can expose new functionality to the JavaScript running in the Titanium SDK. The BlackBerry 10 NDK uses the Cascades framework as its backbone framework, which is built on top of Qt. Therefore, all the BlackBerry components are derived from the QObject class. Instead of the STL containers, like string or map, you will be working with the Qt containers, like QString, QMap, or QList.

This guide covers how to:

  • Create a BlackBerry module project

  • Use Titanium Module APIs in a module

  • Build a module

  • Test a module

This guide assumes you are familiar with native development using the BlackBerry 10 NDK and the QNX Momentics IDE (QDE). For more information about the BlackBerry 10 NDK, see http://developer.blackberry.com/native/documentation/core/.

For samples of using the module APIs, see https://github.com/appcelerator/titanium_mobile_blackberry/tree/master/src/tibb/Modules.

BlackBerry Module Prerequisites

To develop a BlackBerry-based module, the following software is required to build a Titanium application for BlackBerry:

  • Titanium SDK

  • Supported versions of BlackBerry NDK and the Momentics IDE, as described in Installing the BlackBerry NDK

  • Studio or the Titanium Command-Line Interface (CLI) for building and running test applications

Creating a Native Module

Using the CLI

Run the Titanium CLI create command to generate a BlackBerry stub module, including documentation that you can use as a starting point to building your own Titanium BlackBerry extension. The project created is used by the QNX Momentics IDE. You cannot work on the native module in Studio.

titanium create -p blackberry --id <MODULE_ID> -n <MODULE_NAME> -t module

Starting with Release 3.3.0, the Titanium CLI generates a module project containing multiple platforms. The module contains platform folders, for example, blackberry, that contain platform-specific resources and common folders for assets, documentation and examples.

Using Studio

Read the Creating a New Titanium Module guide to find out more about creating modules with Studio. Note that you cannot build the native module with Studio; you need to use QDE or the BlackBerry command-line tools.

Importing the Module to QDE

The QDE is the tool most often used to develop native BlackBerry 10 applications. Although the module stub generated during create has a Makefile that can be utilized by command-line tools, developers often find the Eclipse-based IDE easier to use.

To get started with BlackBerry 10 module development, import the module stub project to QDE:

  1. In the menu bar, select File > Import.... The import wizard opens.

  2. Select Generela > Existing Projects in Workspace, and click Next.

  3. In the Select root directory textbox, enter the path to your BlackBerry module stub project or click Browse and navigate to it.

  4. Click Finish.

QDE sets up a new project based on the stub project. In the project, there is a CPP source file with the same name as your module. This file is the entry point and the startup method is the hook to the BlackBerry SDK. When a module is first loaded into a running Titanium application, the startup method is called.

To build the module stub, right-click the project in the Project Explorer view and select Build Configuration > Build All. This command builds both the ARMv7 binary for devices and x86 binary for the simulator.

BlackBerry uses the V8 JavaScript runtime engine, and the generated stub has some simple example of extending the module by "plugging into" V8. To embedded V8 developers and Node.JS developers the code will be very familiar. Using the stub as a sample, add your own BlackBerry 10 extensions and these will be available to Titanium applications. A good resource for how to work with the V8 runtime is the V8 Embedder's Guide https://developers.google.com/v8/embed.

Titanium APIs for Module Development

Module Lifecycle

To execute code before your module starts, add it to the CreateModule() method in the <MODULE_NAME>ModuleStartup.cpp source file. This method is called during application startup. It is only called once, before your module is actually loaded.

BlackBerryModuleModuleStartup.cpp
#include "BlackBerryModuleModuleStartup.h"
#include "BlackBerryModuleModule.h"
Handle<Object> BlackBerryModuleModuleStartup::CreateModule()
{
fprintf(stderr, "Hello, Titanium!"); // OUTPUT THIS LINE TO THE CONSOLE DURING APPLICATION LAUNCH
return BlackBerryModuleModule::CreateModule(); // DO NOT REMOVE THIS LINE
}

Proxies

A TiProxy is a native class exposed to the JavaScript. You will use some helper macros and specific function calls to hook it up with the JavaScript engine. To create a Proxy, your class must extend the TiProxy class and include the TiCore.h header file. See ExamleProxy.h and ExamleProxy.cpp below for a sample.

Methods and Properties

To expose a method or property to the JavaScript, you need to do four things:

  1. Declare the function in the header file.

  2. Use either the EXPOSE_SETTER, EXPOSE_GETTER or EXPOSE_METHOD macro in the header file, where CLASS is the name of the class and METHOD is the name of the method:

    1. Use EXPOSE_SETTER(CLASS, METHOD) for methods that expect a TiValue but returns void.

    2. Use EXPOSE_GETTER(CLASS, METHOD) for methods that do not expect any parameters but returns a TiValue.

    3. Use EXPOSE_METHOD(CLASS, METHOD) for methods that expect a TiValue and returns a TiValue.

  3. Expose the API to JavaScript in the class constructor using one of the below methods. The API_NAME is the API element that is called by the application, while _METHOD is the method name in the source file prefixed with an underscore ('_'):

    1. Use createPropertySetterGetter(API_NAME, _setMETHOD _getMETHOD) for exposing a property with a setter and getter.

    2. Use createPropertySetter(API_NAME, _METHOD) for exposing a property with a setter only.

    3. Use createPropertyGetter(API_NAME, _METHOD) for exposing a property with a getter only.

    4. Use createPropertyFunction(API_NAME, _METHOD) for exposing a method.

  4. Declare the method function in the source file.

The example below shows how to declare methods, setters and getters.

ExampleProxy.h
// Proxy Definition
#ifndef EXAMPLE_PROXY_H
#define EXAMPLE_PROXY_H
#include "TiCore.h"
class ExampleProxy: public Ti::TiProxy {
public:
// **Required** Macro
CREATE_PROXY(ExampleProxy)
// **Required** Parameter `const char*`
ExampleProxy(const char*);
virtual ~ExampleProxy();
// Setter prop example
void setExampleProperty(Ti::TiValue);
EXPOSE_SETTER(ExampleProxy, setExampleProperty);
// Getter prop example
Ti::TiValue getExampleProperty();
EXPOSE_GETTER(ExampleProxy, getExampleProperty);
// Method exampleMethod
Ti::TiValue exampleMethod(Ti::TiValue);
EXPOSE_METHOD(ExampleProxy, exampleMethod);
};
#endif /* EXAMPLE_PROXY_H */
ExampleProxy.cpp
#include "ExampleProxy.h"
// You MUST pass in the `name` of the constructor to the base class
ExampleProxy::ExampleProxy(const char* name) : Ti::TiProxy(name)
{
// Create a setter and getter for the same property
// Note: the setter and getter *MUST* start with `_`
createPropertySetterGetter("exampleProperty", _setExampleProperty, _getExampleProperty);
/* Or create them individually, first approach is recommended if
* both, setter and getter, exist
*
* createPropertySetter("exampleProperty", _setExampleProperty);
* createPropertyGetter("exampleProperty", _getExampleProperty);
*
*/
// Create a method, it also has to start with `_`
createPropertyFunction("example", _example);
}
ExampleProxy::~ExampleProxy()
{
// delete instatiated pointers
}
void ExampleProxy::initStart()
{
// Called when proxy created from JS and
// not properties have been applied
}
void ExampleProxy::initEnd()
{
// Called when proxy created from JS and
// properties have been applied
}
void ExampleProxy::setExampleProperty(Ti::TiValue value)
{
// Note, some values will be undefined
// Check with:
value.isList();
value.isProxy();
// To get a QString
QString val = value.toString();
// To get a number (double)
double num = value.toNumber();
// To get a boolean
bool flag = value.toBool();
// To get a list
QList<Ti::TiValue> list = value.toList()
// To get a proxy, is this takes a proxy
Ti::TiProxy *proxy = value.toProxy();
// To get the actual JavaScript value
Handle<Value> val = value.toJSValue()
// To get a color, if this takes a color, otherwise returns bb::cascades::Color::Yellow
// #include <bb/cascades/Color>
bb::cascades::Color color = value.toColor();
}
Ti::TiValue ExampleProxy::getExampleProperty()
{
// Warning!!! do not "new" a Ti::TiValue, instatiate it on the stack!!
Ti::TiValue value;
// Caution! use only one of these
// Creates a JS String()
value.setString("some string");
// Creates a JS Number()
value.setNumber(12.54);
// Create a JS Bool()
value.setBool(true);
// Returns a proxy
value.setProxy(this);
// Createa a JS Array() with TiValues
QList<Ti::Value> list;
{
Ti::TiValue value1;
value1.setString("one");
list.append(value1);
Ti::TiValue value2;
value2.setNumber(2);
list.append(value2);
Ti::TiValue value3;
value3.setBool(true);
list.append(value3);
}
value.setList(list);
// Return the created value
return value;
}
Ti::TiValue ExampleProxy::exampleMethod(Ti::TiValue value)
{
// Now you can call functions in this class
// Get the value from `value`
QString str = value.toString();
str.prepend("Something added to ");
// Call local function
// doSomethingWith(str) <---- this would be a local function to this class
// Create a return value
Ti::TiValue returnValue;
returnValue.setString(str);
// Can also be null or undefined
returnValue.setUndefined()
// or
returnValue.setNull()
return returnValue;
}

Events

To fire an event, you need to create an instance of the TiEventParameters, then call the fireEvent(QString eventName, Ti::TiEventParameters params) method, for example:

Ti::TiEventParameters params;
params.addParam("some_string_key", "some_value");
params.addParam("some_int_key", 12);
params.addParam("some_double_key", 12.32);
params.addParam("some_proxy_key", this); // `this` is the proxy
// the `e.source` is included
fireEvent("my_event", params);

To know which event has been added, use the onEventAdded(QString) function. Make sure you add it as a virtual function in the header file.

void ExampleProxy::onEventAdded(QString eventName)
{
if(eventName == "my_event")
{
// do something if needed
}
}

Type Conversions

The TiValue class is a simple, yet powerful, class. It's a tiny bridge between the JavaScript types and the native types. The purpose for this is that you, as a module developer, should not be directly touching the JavaScript engine.

Checking Types

To check what native type is returned from JavaScript use these TiValue functions:

Function

Return Value

isList()

boolean

isNull

boolean

isList()

boolean

isUndefined()

boolean

isProxy()

boolean

isNumber()

boolean

For example, to check the value of a JavaScript array or a TiProxy, first check the object is the correct type, then check the values of that object:

void MyProxy::setMyCustomProperty(TiValue value)
{
// check for JS Array
if(value.isList())
{
QList<TiValue> array = value.toList()
// Now you can loop and get all the values
}
// check for TiProxy
if(value.isProxy())
{
TiProxy* proxy = value.toProxy();
}
}

For example, to get a String from JavaScript:

void MyProxy::setMyCustomProperty(TiValue value) {

Building the Module

Building with the CLI tools

Starting with Release 3.3.0, the Titanium SDK provides a Node.js script to build the module. In the module's blackberry, run the build.js script.

node build.js

This script generates the ARM and x86 binaries, and packages a ZIP file for you in the blackberry directory.

Prior to Release 3.3.0, to build the module using the BlackBerry NDK CLI tools, source the environment file in your BlackBerry NDK or Momentics directory, then build your BlackBerry module project. For example, on Mac OS X:

. /Applications/Momentics.app/bbndk-env_10_2_0_1155.sh
make -C ~/Documents/Titanium_Studio_Workspace/BlackBerryModule/module

Building with QDE

To build the module, right-click the project in the Project Explorer view and select Build Configuration > Build All. This command builds both the ARMv7 binary for devices and x86 binary for the simulator.

Testing the Module in a Titanium Application

Starting with Release 3.3.0, the Titanium SDK provides a build script to automatically package your application. In the module's blackberry, unpack the ZIP file to either the project or Titanium SDK folder. You can also use Studio's Help > Install Mobile Module... option from the menu bar.

In the Titanium application, add the module to your tiapp.xml file and load it in to your application using the require method.

tiapp.xml
<modules>
<module platform="blackberry">MODULE_ID</module>
</modules> 
app.js
var bb_module = require('MODULE_ID');
var bb_proxy = bb_module.createExample();
var str = bb_proxy.example("foo");

Prior to Release 3.3.0, you need to manually package your application. To test your module in a Titanium application, after you have finished building it to generate the ARM and x86 binaries, copy the module's arm, x86 and include folders and the manifest file to a path called modules/blackberry/<MODULE_ID>/<VERSION> to either your Titanium SDK path or to your local project, then require in the module to your application. For example, on Mac OS X, you can create a directory for your module and copy the files to it:

cd ~/Library/Application\ Support/Titanium/modules/
mkdir -p blackberry/<MODULE_ID>/<VERSION>
cd blackberry/<MODULE_ID>/<VERSION>
cp -r ~/Documents/Titanium_Studio_Workspace/<MODULE_NAME>/module/arm .
cp -r ~/Documents/Titanium_Studio_Workspace/<MODULE_NAME>/module/x86 .
cp -r ~/Documents/Titanium_Studio_Workspace/<MODULE_NAME>/module/include header
cp -r ~/Documents/Titanium_Studio_Workspace/<MODULE_NAME>/manifest .
